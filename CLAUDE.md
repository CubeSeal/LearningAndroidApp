# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

An Android app showing Sydney / NSW bus departure times, built on Transport for NSW (TfNSW) GTFS data. Static schedule data is bundled as a prebuilt SQLite database (refreshed from GitHub Releases), and live delays come from the TfNSW GTFS-Realtime feed.

The repo has two independent Gradle projects:
- `Learning/` — the Android app (this is the project you build/run; root `build.gradle.kts` and `settings.gradle.kts` live here, *not* at the repo root).
- `tools/gtfs-converter/` — a standalone JVM program that downloads the TfNSW GTFS bundle and produces the Room-compatible `gtfs.db.gz` published to GitHub Releases.

## Version control: jj, not git

This repo is managed with [Jujutsu (`jj`)](https://jj-vcs.github.io/) on top of a git backend. Use `jj` commands (`jj st`, `jj diff`, `jj log`, `jj new`, `jj describe`), not `git` commands, for normal work. There is no "staging" — the working copy is itself a commit (`@`). Do not run `git commit`/`git add`.

## Commands

All app commands run from inside `Learning/`:

```bash
cd Learning
./gradlew assembleDebug          # build debug APK
./gradlew installDebug           # build + install on a connected device/emulator
./gradlew lint                   # Android lint
./gradlew test                   # JVM unit tests (app/src/test)
./gradlew connectedAndroidTest   # instrumented tests (needs device/emulator)

# run a single unit test
./gradlew test --tests "com.example.learning.ExampleUnitTest.someMethod"
```

The GTFS converter is a separate project with its own wrapper-less Gradle setup:

```bash
cd tools/gtfs-converter
TFNSW_API_KEY=<key> gradle run    # downloads GTFS, builds gtfs.db.gz locally
```

### Dev environment (Nix)

`flake.nix` provides a dev shell with Android Studio, the Android SDK, JDK 17 (sets `JAVA_HOME`, used by the Gradle CLI), and `kotlin-language-server`; `.envrc` (`use flake`) loads it via direnv. The SDK path and `TRANSPORT_NSW_API_KEY` live in `Learning/local.properties` (gitignored — never commit it).

## Secrets

`TRANSPORT_NSW_API_KEY` in `local.properties` is exposed to app code as `BuildConfig.TRANSPORT_NSW_API_KEY` via the secrets-gradle-plugin. The realtime repo (`repos/GTFSRealtime.kt`) reads it to authenticate against the TfNSW realtime feed. The converter reads `TFNSW_API_KEY` from the environment instead.

## Architecture

### Data flow (single direction)

```
Repositories (repos/, db/)  →  TransitInfo  →  ViewModels  →  Compose screens (ui/)
```

- **`ApplicationRepos`** (`ViewModels.kt`) is the manually-wired DI container, held by `LearningApplication`. There is no DI framework — dependencies are constructed by hand and passed down. `MainActivity` requests location permission first, then calls `repos.initAll()`; the UI shows `LoadingScreen` until `repos.loaded` flips true.
- **`TransitInfo`** (`BusInfo.kt`) is the domain layer that merges static schedule + realtime + location + settings into `StateFlow`s that ViewModels consume. It owns the reactive pipeline: `currentMinute` ticks each minute; `closestBusStops`, `focusedBusStop`, `associatedTrips`, `associatedStopTimes`, and `filtersForBusStop` are all derived flows. The realtime overlay is keyed by `(tripId, stopId)` and joined onto static records as a nullable field (`StopTimesRecordWithRealtime`) — null means "no realtime info yet."
- **ViewModels** (`HomeViewModel`, `PickStopViewModel`, `TripsViewModel`) are thin; they mostly re-expose `TransitInfo` flows and hold UI-local state (filter selection, search query). They're created via `AppViewModelProvider.Factory`.

### Navigation

Type-safe Compose Navigation with `@Serializable` route classes defined at the top of `MainActivity.kt` (`Home`, `Trips`, `PickStop`). `HomeNavHost` wires destinations; `NavController.resetTo()` is a helper for tab-style resets.

### Static GTFS database (Room)

- `db/gtfsStaticDB.kt` defines all Room entities (matching GTFS files: agency, routes, calendar, calendar_dates, trips, stops, stop_times) plus `globbed_stops`, the DAO with hand-written join queries, and `GtfsDatabase`. Generated Room code lives under `app/build/generated/ksp/` (KSP).
- The DB is **not** built on-device. It ships prebuilt: `GtfsDatabase.buildDatabase` uses `createFromFile` to seed from a downloaded `gtfs_downloaded.db`, and `replaceWith` hot-swaps a freshly downloaded DB at runtime. Schema version bumps use `fallbackToDestructiveMigration` (the DB is disposable, regenerated by the converter).
- **"Globbing"** groups physically-separate platform stops under one logical stop (e.g. all platforms of a train station). The converter derives `globbed_stops` by parsing `"<Station Name>, Platform N"` style stop names (`createGlobbedStops`). Queries `COALESCE` the globbed id/name with the raw stop when a stop isn't globbed. `GlobbedStopRecord` is the app-level grouped representation.
- **Time handling**: GTFS times can exceed 24:00 (after-midnight services). `parseGtfsDateTime` builds a `LocalDateTime` by adding raw hours/mins/secs to the service date's start-of-day, so "25:30:00" resolves correctly. Service dates come from the pre-materialised `service_dates` table (written by the converter, read by the app via `GtfsDao.getAllServiceDates()`; no runtime date-expansion in the app).

### Static DB sync

`GtfsStaticRepository.syncGtfsDatabase` checks the latest GitHub Release of `CubeSeal/LearningAndroidApp`, compares `published_at` against a saved timestamp file, downloads the `.db.gz` asset, decompresses, verifies the SHA-256 embedded in the release notes, then calls `GtfsDatabase.replaceWith`. The release is produced daily by `.github/workflows/build-gtfs-db.yml` running the converter.

## Testing

JVM unit tests run through Gradle inside the Nix dev shell (which provides JDK 17 + `JAVA_HOME`). There is **no** `nix flake check` / hermetic build — Android + Gradle need network for dependency resolution, which the nix sandbox forbids, so tests are run via the dev shell by design.

```bash
cd Learning
./gradlew testDebugUnitTest                                         # fast inner loop
./gradlew test                                                      # debug + release variants
./gradlew testDebugUnitTest --tests "com.example.learning.HomeViewModelTest"
./gradlew connectedDebugAndroidTest                                 # Room DB tests (needs device)
```

The **humble-object** regime below applies to the **app** (`Learning/`). The converter (`tools/gtfs-converter/`) has its **own hermetic tests** for the build pipeline: the input IO boundary is the `GtfsRowSource` interface (`FileGtfsRowSource` reads CSVs in production; `InMemoryGtfsRowSource` is the test fake), and `buildDatabaseInto(conn, feeds)` builds into an in-memory `jdbc:sqlite::memory:` connection the test queries directly — so the merge/prefix/glob behaviour is asserted state-based with no files. Run them with `Learning/gradlew -p tools/gtfs-converter test` (the converter has no wrapper of its own; borrow the app's). What stays **outside** tests is `main()`'s IO glue — the network download and the `gh`-shelling release — verified manually by running it (`TFNSW_API_KEY=<key> gradle run`) and inspecting the output DB.

### Test boundary rule (humble-object regime)

**ViewModels own all UI state and actions — composables are dumb renderers.** The test drives ViewModels directly as plain JVM objects, never rendering any Compose UI. Robolectric is gone.

**Persistent state = `StateFlow`; one-shot effects (nav, snackbar, scroll-to-top) = `Channel(BUFFERED).receiveAsFlow()` exposed as `Flow`.** Each screen's composable collects these with `LaunchedEffect` and applies them to the NavController or local Compose state. Composables hold no decision state.

**What belongs in JVM tests (`src/test/`)**:
- `HomeViewModelTest`, `PickStopViewModelTest`, `TripsViewModelTest` — drive VMs directly via `runTest(rule.dispatcher)` + Turbine; construct `TransitInfo` from `Fake*Source` doubles with `scope = backgroundScope`.
- `BootstrapGtfsTest` — pure function, no Android deps.
- Canonical pure/library functions (`parseGtfsDateTime`, `transitModeOf`) tested at their own API.

**What belongs in instrumented tests (`src/androidTest/`)**:
- `GtfsValidationTest`, `ServiceDatesReadTest` — use `Room.inMemoryDatabaseBuilder(ApplicationProvider…)` which requires a real Android context. `@RunWith(AndroidJUnit4::class)`, no `@Config`.

> **Do NOT** (anti-patterns that must not come back):
> - Re-introduce Robolectric / `ActivityScenario` / `createEmptyComposeRule` in `src/test/` — the point of humble objects is that this is no longer needed.
> - Poke `TransitInfo` flows directly in a Compose rendering context — use Turbine + VM tests instead.
> - Add `@Config(sdk=[34])` / `@RunWith(RobolectricTestRunner)` anywhere in `src/test/`.

**How VM tests are wired:**

```kotlin
// MainDispatcherRule sets Dispatchers.Main to UnconfinedTestDispatcher — viewModelScope uses it.
@get:Rule val rule = MainDispatcherRule()

@Test
fun test() = runTest(rule.dispatcher) {
    val transitInfo = TransitInfo(fakes…, scope = backgroundScope)
    val vm = HomeViewModel(transitInfo)
    // StateFlow values are immediately available; use Turbine for flow assertions:
    vm.associatedStopTimes.test {
        assertEquals(2, awaitItem().size)
        vm.toggleFilterForBusStops(filter)
        assertEquals(1, awaitItem().size)
        cancelAndIgnoreRemainingEvents()
    }
}
```

`TransitInfo` takes `scope = backgroundScope` (cancelled at test end — no resource leaks). `viewModelScope` uses `Dispatchers.Main` (set to `UnconfinedTestDispatcher` by the rule, so coroutines run eagerly without explicit advancement).

**Fakes live in `src/main`, not `src/test`.** `Fake*Source` classes are colocated in `repos/*.kt` alongside their real counterparts. They ship in the release APK; `src/debug` is the escape hatch if that ever matters. `FakeAppContainer` stays in `Application.kt` for the `AppContainer` seam (still selectable for any manual UI smoke testing).

The four repo seams are plain `interface`s — `StaticGtfsSource`, `RealtimeGtfsSource`, `LocationSource`, `SettingsSource`. Use plain interfaces, not `sealed interface`, so fakes can implement them.

**Fakes model behaviour over a dataset — they are NOT canned-response lookup tables.** A fake is seeded with a small *world* (`FakeStaticGtfsSource(globbedStops, stopTimesRecords)` — a list of real stops and a list of real stop-times) and each method **computes** its answer from that world exactly as production would: `getGlobbedStopById` filters by id, `getGlobbedStopsByName` does a name-substring match, `getNClosestStops` sorts by real Euclidean distance to the passed `LatLon`, `getStopTimesBy{Stop,TripId}` filter the stop-times list. This is a deliberate move for strict TDD:

> **Do NOT** seed a fake with maps keyed by the exact query params it will be asked (the old `stopsById` / `stopsByName` / `closest` / `stopTimesByStop` / `stopTimesByTrip` shape). That makes a method a one-line `map[key]` return, so the test only asserts "the fake gives back what I stuffed in" — it's tautological, exercises no real logic (distance sort, name search, id join), and silently couples the test to the query shape. Seed a **realistic dataset** and let the fake derive answers, so the test drives genuine behaviour and can't cheat.

To keep the fake honest, the conversion helpers `collateToGlobbedStopRecord` and `toStopTimesRecord` are **top-level functions** in `GTFSStatic.kt` (lifted out of the real repository's private methods), so the real repo and the fake collate GTFS rows into domain records through the same code path rather than two hand-kept-in-sync copies.

**TripsViewModel takes explicit typed args** (`tripId`, `stopId`, `date`) rather than a `SavedStateHandle` — `AppViewModelProvider.Factory` calls `savedStateHandle.toRoute()` and passes the parsed values. This keeps `TripsViewModel` free of Android `Bundle` deps and fully JVM-testable.

**Startup provisions then validates.** The DB is never bundled — it only arrives via `syncGtfsDatabase` (downloads the latest GitHub release). So `initAll()` runs `bootstrapGtfs(validate, sync)`: if `validateGtfsDb` (schema + key invariants — tables non-empty, `service_dates` present) is already `Ok` the DB is trusted as-is; otherwise it force-downloads the latest release and re-validates. Only if it is *still* `Invalid` after that sync does it set `loadError`, so `MainActivity` shows an error screen and halts. The provision/validate decision lives in the pure `bootstrapGtfs` seam (tested in `BootstrapGtfsTest`).

- `testOptions.unitTests.isReturnDefaultValues = true` (`app/build.gradle.kts`) makes stubbed `android.*` calls (e.g. `Log`) return defaults instead of throwing in JVM tests.
- `.gitignore` ignores `/test/` (anchored) — the top-level GTFS sample-data dir only; an unanchored `test/` would also swallow `app/src/test`.

## Conventions

- Domain/record data classes consumed by Compose are annotated `@Immutable`. Keep them flat — joins and filtering are done in SQL (see DAO) rather than by nesting/mutating object graphs in Kotlin.
- All async work is coroutines + `Flow`. `TransitInfo` and repos expose `StateFlow`s started with `SharingStarted.Eagerly` or `WhileSubscribed`; ViewModels collect with `collectAsStateWithLifecycle`.
- `applicationId`/namespace is still the template default `com.example.learning`. `minSdk 24`, `compileSdk 36`, Java 11 with core library desugaring (needed for `java.time` on API 24).
- Dependencies are managed through the version catalog at `Learning/gradle/libs.versions.toml`.
