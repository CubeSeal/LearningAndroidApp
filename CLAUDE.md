# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

An Android app showing Sydney / NSW bus departure times, built on Transport for NSW (TfNSW) GTFS data. Static schedule data is bundled as a prebuilt SQLite database (refreshed from GitHub Releases), and live delays come from the TfNSW GTFS-Realtime feed.

The repo has two independent Gradle projects:
- `Learning/` — the Android app (this is the project you build/run; root `build.gradle.kts` and `settings.gradle.kts` live here, *not* at the repo root).
- `tools/gtfs-converter/` — a standalone JVM program that downloads the TfNSW GTFS bundle and produces the Room-compatible `gtfs.db.gz` published to GitHub Releases.

## Version control: jj, not git

This repo is managed with [Jujutsu (`jj`)](https://jj-vcs.github.io/) on top of a git backend. Use `jj` commands (`jj st`, `jj diff`, `jj log`, `jj new`, `jj describe`), not `git` commands, for normal work. There is no "staging" — the working copy is itself a commit (`@`). Do not run `git commit`/`git add`.

## Commands

All app commands run from inside `Learning/`:

```bash
cd Learning
./gradlew assembleDebug          # build debug APK
./gradlew installDebug           # build + install on a connected device/emulator
./gradlew lint                   # Android lint
./gradlew test                   # JVM unit tests (app/src/test)
./gradlew connectedAndroidTest   # instrumented tests (needs device/emulator)

# run a single unit test
./gradlew test --tests "com.example.learning.ExampleUnitTest.someMethod"
```

The GTFS converter is a separate project with its own wrapper-less Gradle setup:

```bash
cd tools/gtfs-converter
TFNSW_API_KEY=<key> gradle run    # downloads GTFS, builds gtfs.db.gz locally
```

### Dev environment (Nix)

`flake.nix` provides a dev shell with Android Studio, the Android SDK, JDK 17 (sets `JAVA_HOME`, used by the Gradle CLI), and `kotlin-language-server`; `.envrc` (`use flake`) loads it via direnv. The SDK path and `TRANSPORT_NSW_API_KEY` live in `Learning/local.properties` (gitignored — never commit it).

## Secrets

`TRANSPORT_NSW_API_KEY` in `local.properties` is exposed to app code as `BuildConfig.TRANSPORT_NSW_API_KEY` via the secrets-gradle-plugin. The realtime repo (`repos/GTFSRealtime.kt`) reads it to authenticate against the TfNSW realtime feed. The converter reads `TFNSW_API_KEY` from the environment instead.

## Architecture

### Data flow (single direction)

```
Repositories (repos/, db/)  →  BusInfo  →  ViewModels  →  Compose screens (ui/)
```

- **`ApplicationRepos`** (`ViewModels.kt`) is the manually-wired DI container, held by `LearningApplication`. There is no DI framework — dependencies are constructed by hand and passed down. `MainActivity` requests location permission first, then calls `repos.initAll()`; the UI shows `LoadingScreen` until `repos.loaded` flips true.
- **`BusInfo`** (`BusInfo.kt`) is the domain layer that merges static schedule + realtime + location + settings into `StateFlow`s that ViewModels consume. It owns the reactive pipeline: `currentMinute` ticks each minute; `closestBusStops`, `focusedBusStop`, `associatedTrips`, `associatedStopTimes`, and `filtersForBusStop` are all derived flows. The realtime overlay is keyed by `(tripId, stopId)` and joined onto static records as a nullable field (`BusStopTimesRecordWithRealtime`) — null means "no realtime info yet."
- **ViewModels** (`HomeViewModel`, `PickStopViewModel`, `TripsViewModel`) are thin; they mostly re-expose `BusInfo` flows and hold UI-local state (filter selection, search query). They're created via `AppViewModelProvider.Factory`.

### Navigation

Type-safe Compose Navigation with `@Serializable` route classes defined at the top of `MainActivity.kt` (`Home`, `Trips`, `PickStop`). `HomeNavHost` wires destinations; `NavController.resetTo()` is a helper for tab-style resets.

### Static GTFS database (Room)

- `db/gtfsStaticDB.kt` defines all Room entities (matching GTFS files: agency, routes, calendar, calendar_dates, trips, stops, stop_times) plus `globbed_stops`, the DAO with hand-written join queries, and `GtfsDatabase`. Generated Room code lives under `app/build/generated/ksp/` (KSP).
- The DB is **not** built on-device. It ships prebuilt: `GtfsDatabase.buildDatabase` uses `createFromFile` to seed from a downloaded `gtfs_downloaded.db`, and `replaceWith` hot-swaps a freshly downloaded DB at runtime. Schema version bumps use `fallbackToDestructiveMigration` (the DB is disposable, regenerated by the converter).
- **"Globbing"** groups physically-separate platform stops under one logical stop (e.g. all platforms of a train station). The converter derives `globbed_stops` by parsing `"<Station Name>, Platform N"` style stop names (`createGlobbedStops`). Queries `COALESCE` the globbed id/name with the raw stop when a stop isn't globbed. `GlobbedBusStopRecord` is the app-level grouped representation.
- **Time handling**: GTFS times can exceed 24:00 (after-midnight services). `parseGtfsDateTime` builds a `LocalDateTime` by adding raw hours/mins/secs to the service date's start-of-day, so "25:30:00" resolves correctly. Service dates come from `calendar` expanded into per-date sequences (`preloadCalendarDates`, cached with a mutex, capped at +2 weeks).

### Static DB sync

`GtfsStaticRepository.syncGtfsDatabase` checks the latest GitHub Release of `CubeSeal/LearningAndroidApp`, compares `published_at` against a saved timestamp file, downloads the `.db.gz` asset, decompresses, verifies the SHA-256 embedded in the release notes, then calls `GtfsDatabase.replaceWith`. The release is produced daily by `.github/workflows/build-gtfs-db.yml` running the converter.

## Testing

JVM unit tests run through Gradle inside the Nix dev shell (which provides JDK 17 + `JAVA_HOME`). There is **no** `nix flake check` / hermetic build — Android + Gradle need network for dependency resolution, which the nix sandbox forbids, so tests are run via the dev shell by design.

```bash
cd Learning
./gradlew testDebugUnitTest                                         # fast inner loop
./gradlew test                                                      # debug + release variants
./gradlew testDebugUnitTest --tests "com.example.learning.BusInfoTest"
```

The **red-green-refactor** regime below applies to the **app** (`Learning/`). The converter (`tools/gtfs-converter/`) has its **own hermetic tests** for the build pipeline: the input IO boundary is the `GtfsRowSource` interface (`FileGtfsRowSource` reads CSVs in production; `InMemoryGtfsRowSource` is the test fake), and `buildDatabaseInto(conn, feeds)` builds into an in-memory `jdbc:sqlite::memory:` connection the test queries directly — so the merge/prefix/glob behaviour is asserted state-based with no files. Run them with `Learning/gradlew -p tools/gtfs-converter test` (the converter has no wrapper of its own; borrow the app's). What stays **outside** tests is `main()`'s IO glue — the network download and the `gh`-shelling release — verified manually by running it (`TFNSW_API_KEY=<key> gradle run`) and inspecting the output DB.

The app follows **red-green-refactor** with **state-based fakes**, asserting behaviour through public APIs (never call-count/interaction assertions):

- The four repositories implement plain `interface` seams — `StaticGtfsSource`, `RealtimeGtfsSource`, `LocationSource`, `SettingsSource` — each colocated in its `repos/*.kt` file. `BusInfo` depends on these interfaces, not the concrete classes. Use plain interfaces for DI seams; reserve `sealed` for closed variant hierarchies (e.g. `BusFilterOptions`). A `sealed interface` can't be implemented from the `src/test` compilation, so it would block fakes.
- The domain is framework-free: the location seam exposes the domain `LatLon`, not `android.location.Location`, so `BusInfo` tests are plain JVM tests (no Robolectric).
- Hand-written state-based fakes live in `app/src/test/java/com/example/learning/repos/` (`Fake*Source`), backed by `MutableStateFlow`/in-memory maps and seeded per test.
- Test libs: JUnit4 + `kotlinx-coroutines-test` (`runTest`, `backgroundScope`) + Turbine (`flow.test { awaitItem() }`).
- `testOptions.unitTests.isReturnDefaultValues = true` (`app/build.gradle.kts`) makes stubbed `android.*` calls (e.g. `Log`) return defaults instead of throwing in JVM tests.
- Copy from the worked examples: `ParseGtfsDateTimeTest` (pure function) and `BusInfoTest` (fakes + Turbine).
- `.gitignore` ignores `/test/` (anchored) — the top-level GTFS sample-data dir only; an unanchored `test/` would also swallow `app/src/test`.

## Conventions

- Domain/record data classes consumed by Compose are annotated `@Immutable`. Keep them flat — joins and filtering are done in SQL (see DAO) rather than by nesting/mutating object graphs in Kotlin.
- All async work is coroutines + `Flow`. `BusInfo` and repos expose `StateFlow`s started with `SharingStarted.Eagerly` or `WhileSubscribed`; ViewModels collect with `collectAsStateWithLifecycle`.
- `applicationId`/namespace is still the template default `com.example.learning`. `minSdk 24`, `compileSdk 36`, Java 11 with core library desugaring (needed for `java.time` on API 24).
- Dependencies are managed through the version catalog at `Learning/gradle/libs.versions.toml`.
